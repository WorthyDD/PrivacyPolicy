核心结论

2026-06-11 前后，前端 v1-revamp 仓库中多项变更叠加，引发 WebView chunk 错误死循环 + Discover 加载卡死， 导致 hybrid_process_terminated (OOM) 飙升 2 倍以上，同时 feed card expose 数量急剧下降。 iOS native shell (2026-06-16 上线) 逐步替换 H5 后，OOM 率从 9.48% 峰值持续下降至 3.09%。

3. 根因链 (修订版)

修正: Discover 修复不影响 Feed 曝光 (用户打开 App 首先进入 Feed Tab,不是 Discover)

06-12 的 Discover 双修复 (pull-to-refresh + load issue) 主要缓解了 Discover 页面的内存泄漏, 对 Feed 曝光恢复贡献有限。OOM 暴涨的真正触发点是 Feed 层面的变更。

真正的根因链

06-05: Feed API fallback 上线 (#1971)
  └── 当 Flutter feed 返回空数据时,自动走 direct API 兜底
  └── 增加了额外数据加载路径 → 内存峰值 + 并发请求

06-08: Feed Card Analytics Context 上线 (#1978)
  └── 为每张 feed 卡片注册 requestId / cardType / isOfflineCards
  └── 227 行新代码,React Context 嵌套层级增加
  └── 每张卡片的 expose 埋点携带更多引用 → 闭包链延长 → 内存不释放

为何 06-08~06-10 OOM 反而下降了?

analytics context (06-08 部署) 和 API fallback (06-05 部署) 在 CDN 全量生效需要 1-2 天。 06-08 ~ 06-10 的下降是因为旧代码还在线上运行,新代码尚未覆盖全部用户。 到 06-10 晚间,新代码基本覆盖完毕;06-11 起 chunk fix 叠加生效,三重压力同时出现,导致 OOM 飙升。

5. 关键结论

  1. OOM 飙升 (4.4% → 9.5%) = 三项 Feed 层变更叠加: (1) c6b3a2a7 API fallback 双数据源 (2) 341344df 每卡片 analytics context 阻止 GC (3) f28d19bf chunk fix 的 cache-bust replace() 创建新 WebView 实例。三项独立变更在 06-10 晚间首次同时覆盖到全部用户,06-11 集中爆发
  2. Discover 修复 (06-12) 对 Feed 曝光恢复贡献有限 — 用户默认进入 Feed Tab,Discover 的 pull-to-refresh 修复和加载死锁修复不影响 Feed 的内存泄漏
  3. 06-08~06-10 OOM 下降是 CDN 部署延迟导致的假象 — 新代码尚未覆盖全部用户,到 06-10 晚间覆盖率才到位
  4. Native shell (06-16) 根本性改善 — 实验组 (ios_native) OOM 率仅 2.4-3.6%,对照组 (h5) 高达 7-11%,差距约 3 倍。Native shell 替换 H5 是唯一的根本解决方案
  5. app_launch 污染 (06-20 起) 是 Feed 到达率继续下滑的另一独立原因 — 与 OOM 问题无关,需单独处理

6. 建议

  • 短期: 将 Feed 到达率的分母从 app_launch 切换为 app_foreground (不受 lifecycle_state 污染), 并配合 1.72.5+ 版本过滤, 以获取干净的到达率指标
  • 中期: 在前端 CI 中增加 memory regression test (尤其是 G3 game + Discover + Feed 混合场景), 防止重构再次引入内存泄漏
  • 长期: 加速 iOS native shell 全量替换 H5, 当前数据已证明 native shell 内存稳定性远优于 H5
